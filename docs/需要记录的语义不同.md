### 语义不一致（主要是“传输形态”差异）

- `getProviderLimitUsageBatch()`：
  - TS 直接返回 `Map<number, ProviderLimitUsageData>`（Server Action 内部结构，不经 JSON）。
  - KarisCode（HTTP）返回 `[[providerId, data], ...]`（Map entries 的 JSON 可表示形式），调用方需自行还原为 Map/对象。
- `getProviderLimitUsage()` / `getProviderLimitUsageBatch()` 的 `resetAt`：
  - TS 侧是 `Date`（Server Action 可传递）。
  - KarisCode 侧输出为 ISO8601 字符串（JSON 传输所需）。

### 语义不一致

- **传输形态（Date）**：
  - TS `UsageLogRow.createdAt` 是 `Date | null`
  - KarisCode `createdAt` 输出为 ISO 字符串或 `null`（HTTP+JSON 必然差异）

- **传输形态/格式化（Date、本地化、字段缺省）**
  - TS 返回多处 `Date`（例如 `user.expiresAt`、key `createdAt`、`getUserLimitUsage.dailyCost.resetAt`）；KarisCode 对应字段多为 RFC3339/ISO 字符串。
  - TS `getUsers` 中部分文案依赖 `next-intl`（例如 key `neverExpires`、`createdAtFormatted` 的 locale 格式化）；KarisCode 侧使用固定中文/固定时间格式（如 `"永不过期"`、`"2006-01-02 15:04:05"`），不随 locale 变化。
  - TS 与 KarisCode 的“null/undefined”在 HTTP+JSON 中只能近似表达，且不包含 Next.js 的 RSC 序列化能力（Date 可被直接还原为对象）。
